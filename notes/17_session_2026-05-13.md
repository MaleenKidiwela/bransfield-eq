### Run 1: `--cache-size 1500` (~52 GiB cap)
- Launched 06:53; **no anchor checkpoint logged in 31 min**. Pod memory hit 185/187 GiB (99%).
- py-spy was blocked by `ptrace_scope=1`; used `/proc/<pid>/task/*/wchan` instead — 3 threads in `D` state on `wait_on_page_bit_common`/`do_user_addr_fault` → **page-cache thrashing**. Process was forward-progressing in raw CPU terms but bouncing on disk for evicted page-cache.
- Killed and relaunched with `--cache-size 600`.

### Run 2: `--cache-size 600` (~20 GiB cap)
- Launched 07:39. RSS dropped to ~46 GiB; but D-state threads on disk I/O went **up** to 7 (more re-reads).
- Ran 4h 52min unattended → 150 / 36,757 anchors done. Effective rate ~0.01 anchors/sec. **ETA ~1,000 hours (42 days)** — won't finish.
- Pod memory back at 184/187 GiB by hour 5 (page cache filling reclaimable but the run is still slow).

### Root cause (algorithmic, not memory)
Each anchor pulls ~80 partner events; each partner needs ~5-10 station-day cache entries. With 32 threads working anchors in arbitrary order, the *collective* working set is thousands of unique station-days — neither cap (1500 or 600) holds them all hot. So LRU evicts entries that other threads are about to touch. Neither cache size fixes this.

### Real fix (not yet applied)
**Sort anchors by anchor-event origin time** (or by spatial cluster + time) before submitting to the ThreadPoolExecutor. Adjacent-in-time anchors share station-days, giving the LRU real locality. Expected ~5-line change in `main()` of `18_growclust_xc_prep.py`:

```python
anchors = sorted(pairs_by_anchor.keys(),
                 key=lambda i: events.iloc[i].origin_time)
```

Then 32 workers all stay roughly inside the same date window → cache hits dominate.

## Update — date-sort fix tried (Run 3, `--cache-size 600` + anchor sort by origin time)

Patched `main()` in `scripts/18_growclust_xc_prep.py` to (a) sort each anchor's partner list by partner origin time and (b) sort all anchors by anchor origin time before submitting to `ThreadPoolExecutor`. Idea was to give the LRU temporal locality so 32 threads stay inside a tight time window.

Measured first checkpoint: **50 anchors in 1435 s (~24 min) = 0.035 anchors/sec → ETA ~12 days**. Compared to the unsorted run's ~42 days, that's a **real 4× speedup** — but still nowhere near usable. Rate further dropped to ~0.028/s by 53 min in.

Thread-state breakdown after sort fix: 28 R, 4 D (was 24 R, 7 D before sort) — confirms fewer threads stalling on disk page faults, but the underlying cost structure is the same.

**Root cause confirmed at architectural level, not tuning level:** `_process_anchor` re-decodes mseed inside the pair loop. Even with cache hits, every pick gets a fresh `slice_window_from_array` call (mean-subtract + L2-norm), and every cache miss pays a full obspy parse + resample. At ~30 partners × ~10 picks per anchor × 5–10 shared phases per pair, the per-anchor cost stays at ~36 s wall regardless of cache size or sort order.

## Plan for next step — pre-window every pick once into HDF5/memmap

Cross-checked against published practice (EQcorrscan `catalog_to_dd.write_correlations`, hypoDDpy, QuakeSupport, the Rutford icequake catalog workflow). **Universal pattern: pre-window every (event, station, phase) pick into a flat array, then XC the snippets — never decode mseed inside the pair loop.**

Math justifying the pivot:
- 483,516 picks × (±1.5 s × 100 Hz × float32) = **~580 MB total**. Fits in a single `numpy.memmap`.
- Working-set reduction vs current station-day cache: ~50,000×. The 187 GiB pod cap becomes irrelevant.
- Per-anchor cost drops from ~36 s (current measured) to ~0.3 s (theoretical: 300 array fetches + 150 FFT XCs of 300-sample windows). Realistic 50–100× speedup.
- **Expected total wall time: ~6–8 hours end-to-end** (≈20 min parallel pre-window pass + ~6–7 h vectorized XC pass).

Implementation outline (committed kill of Run 3 first):

1. **New script `scripts/18b_prewindow_picks.py`** — pre-allocate `growclust/picker_only/pick_windows.npy` (memmap, shape `[n_picks, 300]`, float32). For each station-day file (parallel, 32 thread workers), load + resample once, then write each pick's ±1.5 s window into its row by `pick_id`. Workers touch disjoint rows → no locking. Also writes a sidecar `pick_metadata.parquet` with `pick_id`, `event_idx`, `sta_key`, `phase`, `pick_time` for the XC stage to index by.
2. **Refactor `scripts/18_growclust_xc_prep.py`** — replace the `_get_wf` + `slice_window_from_array` path with a direct `pick_windows[pick_id]` lookup. Keep the existing pair selection (5 km / 80-per-event already gets pair count to 1.1M; DBSCAN pre-clustering would only shave another ~30–50%, not worth it now). Vectorize FFT XC over batches of pairs sharing a (station, phase) where possible.
3. **`scripts/20_pipeline_finale.sh`** — insert the pre-window stage between Stage 2 (pyocto merge) and Stage 3 (XC).

Pod memory needs collapse to <10 GiB anon. **No admin email needed unless pre-windowing hits an unexpected wall** — hold the previously-drafted memory-bump request until after the rewrite is tested.

## Issues encountered (chronological)

1. **Pod OOM (Run 1, cache 1500 / ~52 GiB).** Pod hit 185 / 187 GiB within 30 min; 0 anchor checkpoints logged. `dmesg` empty (cgroup OOMs don't log to host kernel). `memory.events.oom_kill = 0` after pod restart is misleading — the counter resets. Symptom was the "Server unavailable — restart it?" JupyterHub banner, same as the original crashes.
2. **`nohup` does not survive pod death.** Once the pod is OOM-killed by the cgroup, every process inside dies regardless of `nohup`/`setsid`. Only mitigations are out-of-pod (SLURM, etc.) or bounding in-process memory. SLURM was confirmed unavailable on this hub.
3. **No swap.** `memory.swap.max = 0` → cgroup OOM fires *immediately* on the offending allocation when `memory.max` is exceeded. No grace period.
4. **Page-cache thrashing (Run 1).** Process anon RSS ~80 GiB + kernel file-cache ~115 GiB pushed total cgroup memory to 99% of cap. 3+ threads stuck in `D` state on `wait_on_page_bit_common` / `do_user_addr_fault` — threads forward-progressing in raw CPU terms but bouncing on disk for evicted page-cache pages.
5. **`ptrace_scope = 1`.** `py-spy dump --pid <pid>` denied permission. No sudo. Fallback: `/proc/<pid>/task/*/wchan` to infer thread states.
6. **Smaller cache didn't fix wall-clock (Run 2, cache 600 / ~20 GiB).** RSS dropped from 80 GiB to 47 GiB, D-state went 3 → 7 (more re-reads, not fewer), and **150 / 36,757 anchors finished in 4 h 52 min → ETA ~42 days**. Per-anchor cost was ~117 s.
7. **Sort-by-origin-time helped, but not enough (Run 3, cache 600 + anchor sort).** Real 4× speedup: first 50 anchors in 1435 s (~24 min) = 0.035 anchors/sec → ETA ~12 days. Rate then drifted to ~0.028/s. Still wall-clock-infeasible.
8. **Architectural root cause:** mseed parse + resample inside the pair loop, even with perfect caching. `slice_window_from_array` is called per (pick, partner) — ~300 calls per anchor — and every cache miss is a full obspy `read()` + `resample()`. No LRU tune can fix that.
9. **`memory.current` is misleading.** Includes reclaimable kernel page cache. Watching `memory.current / memory.max` overstates real pressure; what actually matters for OOM risk is `RssAnon` from `/proc/<pid>/status`.
10. **Bash history truncated.** Only the most recent ~10 commands survive in `~/.bash_history` (JupyterHub login-shell config), so reconstructing how the failed runs were launched relied on log timestamps rather than command history.

## GrowClust non-determinism — root cause identified

The pre-window vs front-load comparison (Jan-Mar 2019) revealed that even with **bit-identical pair content** (verified by canonical-sorted dt.cc diff: 115,217 pairs, 0 differing observation rows), the final relocations disagreed for ~23% of events (p95 |Δh| = 2.81 km, max 4.9 km).

Research into the GrowClust Fortran source (`SRC/growclust.f90` + `SRC/stats_subs.f90`) pinpointed the cause:

- Main loop sorts pairs by `rfactor = ngood * rgoodavg` using `INDEXX` from `stats_subs.f90` — the Numerical Recipes heapsort, which is **not stable**.
- Ties in `rfactor` (common with OBS data: integer `ngood` counts × coarse mean CC) → sort tie-break order is the dt.cc input order.
- Our XC kernel uses `ThreadPoolExecutor.as_completed`, so dt.cc pair-block order is thread-finish order → **non-deterministic across runs**.
- Same data, different dt.cc order → different cluster topology → ~20-25% of events relocate differently. Not documented in the GrowClust paper, user guide, or GitHub issues.

**Implication for the 30-day-vs-yearly comparison:** the p95 |Δh| ≈ 2.84 km tail we saw is partly real science (yearly run has 3.6× more pairs adding constraints) and partly this non-determinism noise floor. We can't cleanly separate the two without deterministic dt.cc.

## GrowClust reproducibility — found the real cause of May-12-vs-today divergence

User pointed out that `growclust/partial30days/dt.cc` (the May 12 dt.cc) was still on disk. I had previously and incorrectly claimed it was overwritten. **It was preserved** along with its `evlist.txt`, `stlist.txt`, and `run/` subdirectory.

Re-running Stage 3 (XC prep) on jan2019 today and canonical-content-diffing against May 12's dt.cc revealed:

- Same **109,574 pair blocks** in both files (0 only-in-A, 0 only-in-B).
- But **100,286 of those pairs (91.5%) have differing observation values.**
- The differences are systematic: every observation on a given pair shifts by the same offset (e.g. 0.0096 s on pair (4946, 9340) -- the same shift on BRA20-P, BRA26-P, BRA26-S).

Same shift across all stations of one pair = the value of `origin_a − origin_b` for that pair. Confirmed by `git show 57c92c9` (the only post-May-12 commit on this script):

```python
-        rows.append((pa.sta_key, pa.phase, pick_dt_obs, cc))   # May 12 (wrong)
+        tt_dt = pick_dt_obs - origin_dt
+        rows.append((pa.sta_key, pa.phase, tt_dt, cc))         # today (correct)
```

**The May 12 dt.cc stored pick-time differentials instead of travel-time differentials.** GrowClust expects `tt_a − tt_b` (travel-time differential) where `tt = pick − origin`; May 12 left the origin-time difference baked into the dt values, contaminating the relative-position signal. Today's code subtracts it.

**Implication:** the May 12 30-day partial catalog (`catalogs/growclust_partial30days.csv`, `notes/figures/growclust_relocations.png`) is based on incorrect dt.cc values. It should be treated as **superseded**, not as a trusted reference. Today's runs are the correct baseline.

Three same-day Jan 2019 reruns confirm this:

| Run | Relocated | Top cluster | Identical to May 12 | p95 \|Δh\| vs May 12 |
|---|---|---|---|---|
| May 12 partial30days | 5,036 | **2,556** | — | — |
| Today A (pre-window) | 5,046 | 2,963 | 56.8% | 2.72 km |
| Today B (front-load) | 5,051 | 2,842 | 56.6% | 2.71 km |
| Today C (re-make pre-window) | 5,050 | 2,899 | 56.7% | 2.80 km |

Today's three runs cluster together (~10-15% intra-day non-determinism) and consistently lean *away* from May 12 in the same direction -- if it were just non-determinism, today's three would scatter around May 12. They don't. The systematic offset is the dt accounting fix.
