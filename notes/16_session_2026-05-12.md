## Pyocto: from 16h monolithic failure → daily chunks in parallel

**Failed approach** (replicated from yesterday's stuck run): single `associate()` call on the whole year, 8 threads, hit 16h with zero output written. Was killed.

**Working approach**: daily chunks running in parallel (`scripts/17e_pyocto_daily_year.sh`):
- 1 chunk per day, 10 threads each, 10 chunks at a time (100 cores per batch)
- 425 days total → 43 batches of 10 days each
- Each batch runs in parallel; next batch only starts when current finishes
- Resume-capable: skips any day already in `catalogs/pyocto_daily/events_YYYY-MM-DD.csv`
- Per-day output written immediately on completion — full visibility, no all-or-nothing risk

**Per-day timing** (post-OBS-online, n≥5 stations):
- Quiet day: 5–10 min
- Dense swarm day (e.g., 2019-01-23 with 1900+ events): 30–45 min
- Whole-year extrapolation: ~22h wall

**Boundary safety**: each chunk loads picks from `[day_start − 120s, day_end + 120s]` (`--margin-seconds 120`) so events near midnight see their late-arriving picks. After `associate()`, events whose origin time falls outside the strict day are filtered out so the adjacent chunks don't double-count them. Confirmed working via Jan 9 chunk's log:

```
strict window: [2019-01-09 00:00, 2019-01-10 00:00)
load window:   [2019-01-08 23:58, 2019-01-10 00:02)  (+120s margin)
margin filter: kept 9/9 events in strict window
```

**Filename pre-filter optimization**: the original `load_picks_for_pyocto()` scanned all 28,000 year-long CSVs for every chunk and filtered by timestamp. Now it only opens CSVs whose filename's day-of-year falls inside the load window — ~76 files per chunk instead of 28,000. Brought per-day startup from ~10 min to <10s.

## Pyocto bug catalogue (all fixed)

`scripts/17_pyocto_associate.py`:

1. **Missing `id` column on stations** — pyocto ≥0.6 requires `id` on the stations DataFrame in addition to the `station`/`network` columns. Added `stations["id"] = stations["station"]` before `associator.associate(...)`.
2. **Missing `transform_stations()` call** — pyocto needs local x/y/z columns projected via the associator's CRS, set by `associator.transform_stations(stations)`. Old script omitted this and failed with `Columns 'x', 'y', 'z' expected ...`.
3. **`Pick` constructor expects float epoch seconds, not pd.Timestamp** — newer pyocto C++ binding signature is `Pick(int, float, str, str)`. Was passing `picks_in["time"]` as datetime which fails. Fix: `picks_in["time"] = picks_in["time"].astype("int64") / 1e9`.
4. **Eikonal solver `EikonalExtError: unexpected/untested code executed`** — caused by `Vs = 0` in the water layer (pyrocko's solver divides by Vs). Set water Vs = 0.5 km/s (physically meaningless but safe — rays from sub-seafloor sources never traverse water, so Fermat routes them through faster rock anyway).
5. **`--label` choice restricted** — original script's `argparse choices=["picker_only", "with_manual"]` rejected arbitrary chunk labels like `monthly_2019-01`. Loosened to free-form, still triggers `with_manual` behaviour only on that exact label.

Also added:
- Timestamped progress logging via `_log()` helper + `python -u` for unbuffered output
- Per-phase wall-time tracking (pick load / model build / associator / associate)
- Exception printing with full traceback before exit
- Pre-flight stats: per-station pick counts, station z range, pick time-range sanity checks

## GrowClust XC prep launched (Stage 3)

Started in parallel with pyocto's daily run on 30 days of partial catalog (`catalogs/pyocto_events_partial30days.csv`, 11,358 events, 154,168 picks). Two more bugs found here before it would run correctly:

1. **`pd.to_datetime` without `unit='s'`** — pyocto saves `time` as float epoch-seconds. `pd.to_datetime(float, utc=True)` defaults to interpreting numeric input as nanoseconds → every event ended up on 1970-01-01. Waveform path lookup then targets `*.1970.001.mseed` which doesn't exist → `kept 0 pairs / 0 obs` for ~10 min before we caught it. Fix: detect numeric dtype and force `unit="s"`.
2. **`ProcessPoolExecutor` imported but never used** — `--workers 24` arg was a no-op; the cross-correlation loop ran single-threaded. Rewrote the loop with `ThreadPoolExecutor(max_workers=args.workers)` over `_process_anchor(anchor)` workers. Threads are fine here because the work is I/O-bound (mseed reads) + numpy XC (releases GIL).

These two were *silent-correctness failures* — script printed normal progress lines while producing useless output. Updated the monitor's grep filter to flag `kept 0 pairs` and `events: 0` patterns so similar bugs don't waste time again.

Status of XC prep launch: 309,641 unique event pairs identified within 5 km, 24-thread cross-correlation running, ~3-4 cores effectively busy (I/O dominated). RSS 16 GB (waveform caches per anchor).

## Stage 4 (GrowClust) — full pipeline working

Ran GrowClust on the 30-day partial XC output. Five bugs uncovered in the runner
(`scripts/19_run_growclust.py`) before completion:

1. **Control file passed via stdin instead of argv[1]** — GrowClust's `growclust.f90:182`
   reads `call getarg(1, infile_ctl)`, so it expects the control file as a command-line
   argument. Fixed: `subprocess.run([args.binary, ctl_path.name], cwd=run_dir, ...)`.

2. **Station list format incompatible** — `NW.STNAME` (period at column 3) fell into
   GrowClust's STNAME-only branch which truncates at 5 chars, then tried to parse the
   remaining char as a real number ("Bad real number in item 1 of list input"). Fix:
   strip network prefix from both `stlist.txt` and `dt.cc` so station codes are bare
   (e.g., `BRA15`, not `ZX.BRA15`).

3. **Travel-time table distance too small** — `tt_del1=150 km` but the network aperture
   is ~165 km (5M.TOW to far events). Bumped `tt_del1` to 300 km and `delmax` to 250 km.

4. **Travel-time table depth too shallow** — events at ≤40 km with `tt_dep1=30 km`.
   Bumped `tt_dep1` to 60 km (matches pyocto's z_max+headroom). Also extended
   `configs/velocity_model.csv` with a half-space marker at 70 km so the vzmodel covers
   the table range.

5. **Earlier `dt.cc` bug surfaced via inputs** — the differential times were raw
   pick-time differences (median |dt| = 21 hours instead of sub-second). Fixed in
   `scripts/18b_fix_dtcc.py` (post-processes existing dt.cc) and in
   `scripts/18_growclust_xc_prep.py:collect_pair_dts` (subtracts origin-time difference)
   for future runs.

After fixes, Stage 4 wall time is **negligible** (<1 s for the relocation itself; the
travel-time table build dominates at ~30 s). Output: `catalogs/growclust_partial30days.csv`.

**Quality of 30-day partial relocation:**

## GrowClust XC prep — silent-correctness bugs

Two upstream bugs in `scripts/18_growclust_xc_prep.py` cost ~2 hours each before being
found:

1. **`pd.to_datetime` without `unit='s'`** — pyocto saves `time` as float epoch-seconds;
   `pd.to_datetime(numeric, utc=True)` defaults to ns, so every event ended up on
   1970-01-01. Waveform path lookup then targeted nonexistent `*.1970.001.mseed` files
   → cache empty → `kept 0 pairs / 0 obs` for 10+ minutes before we noticed. Fix: detect
   numeric dtype and force `unit="s"`.

2. **`ProcessPoolExecutor` imported but never used** — `--workers 24` was a no-op; the
   cross-correlation loop ran single-threaded. Wrapped the loop in
   `ThreadPoolExecutor(max_workers=args.workers)` over `_process_anchor(anchor)` workers.
   Threads (not processes) are fine because the work is I/O-bound (mseed reads) +
   numpy XC which releases the GIL.

Additional efficiency rewrite: **preload every station-day mseed file once** into
numpy arrays at 100 Hz (`preload_all_station_days`), so the inner XC loop is pure
numpy slicing instead of repeated `obspy.read()`. Initial run before this rewrite
was projecting ~4 days at 24 threads; after rewrite, **67 min** total wall time
(20 min preload + 47 min XC) for the 30-day partial with 309k unique event pairs.
