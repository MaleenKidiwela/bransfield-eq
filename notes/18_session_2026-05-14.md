# Session 2026-05-14 — hypoDD at scale: monolithic vs. pruned + sub-cluster

## State at start of day

- Year-long hypoDD v2 run launched 2026-05-13 23:21 UTC. ph2dt produced 302 MB `dt.ct` on the 42,040-event catalog. Two hypoDD binaries running in parallel against it:
  - `picker_only` (gold-standard): `niter_ct=2`, NSET=3, full LSQR schedule.
  - `picker_only_fast`: `niter_ct=1`, NSET=2.
- Both used the v2-fixed pipeline (per-layer Vp/Vs via IMOD=1, calibrated PhaseNet-prob weights, station elevations from `elevation_m`, ph2dt MAXOBS=80, MAXDIST=100 km, NaT guard, depth-sanity post-filter).
- Jan 2019 dry-run finished in 9 min and looked correct, so the v2 config was validated at small scale before scaling up.

## What happened

### Discovered the monolithic run won't converge in any human timescale (~13:30 UTC)

After ~7 h elapsed (fast) and ~6 h (gold), neither run had produced an iteration banner. The hypoDD log was 4,930 lines of "negative depth" warnings from inside iteration 1, then nothing. Process states were healthy (`R`, 99.9 % CPU, stable RSS). The LSQR had been chewing on cluster 1 (20,036 events × 8.5 M dt observations) for hours and had not finished its first matrix-vector product cycle. Extrapolating from "still iter 1 after 11 h" gave a finish window of 3-7 days.

### Bug discovered in `scripts/24_run_hypodd.py`

`subprocess.run` had a 10,800 s (3 h) timeout. That was way too tight for year-long scale and killed the original gold-standard run cleanly mid-iteration. Patched: removed the timeout entirely. The fast run was still alive at the time -- to escape the same fate I `kill`-ed only its python parent, leaving the hypoDD binary orphaned to init (PPID=1). Wrote `scripts/watch_and_finalize_fast.sh` to wait on the orphaned PID and parse the reloc file when (if) it finished. Relaunched the gold-standard with the patched (no-timeout) script.

### Expert consultation

Drafted a focused question for seismologists who have run hypoDD at this scale. Got back three concrete recommendations:

1. **Stock serial hypoDD on 20 k events / 8 M dt is genuinely not pleasant.** Graph density (millions of sparse rows per LSQR iteration on an 80 k-unknown system) dominates beyond a certain event-pair count.
2. **Partitioning is defensible** -- but prefer event-pair graph partitioning (edge weights = number of dt obs, mean CC, distance, shared stations) with overlapping halos / bridge events rather than pure geographic chopping. Yano et al. used 1,257 grid squares for Japan.
3. **No drop-in MPI hypoDD fork.** `relocDD-py` and `FDTCC` cover XC generation, not the LSQR solver itself.

Concrete workflow the expert recommended: **pruned global relocation first, then dense subcluster relocations.**

### Killed both monolithic runs

After 11 h+ each with no progress past iteration 1, killed `picker_only` (PID 23679) and `picker_only_fast` (PID 22568). Memory freed cleanly.

### What stands out

**1. Stage A's RMS is dramatically lower than every other run** — 147 ms vs ~190-200 ms for every other catalog. That's the expected effect of a strict prune: by requiring `MINLNK=16, MINOBS=16, MINWGHT=0.5`, only the highest-coupling pairs survive, and the LSQR converges to a much tighter fit. Stage A is essentially the "high-confidence skeleton" of the year-long catalog — 676 events at sub-100 ms TT-residual precision.

**2. Both Stage B catalogs put the swarm centroid in essentially the same place** (−62.42, −58.38), within ~100 m of each other and ~5 km southeast of the OBS array centre. Variant differences are in event-count (5,064 vs 3,579) and depth distribution (median 3.61 vs 3.78 km), not in the location of the swarm itself.

**3. Stage B depth distributions are narrower than Jan 2019 v2's.** Both Stage B variants have p5–p95 depth spans of ~4-5 km, vs Jan 2019 v2's 7 km span. The strict sub-region partitioning suppresses the long depth tail — events that would have been pulled into deep poorly-constrained positions in a global solve stay closer to the cluster centroid.

**4. Stage B-jan2019 (variant) is *more conservative*** than Stage B (Stage A-seeded): 3,579 events vs 5,064. That's because the variant used tighter regions (1.5 km max radius + 0.8 km buffer) vs 2.0 + 1.0 km. Per-region geometry is more compact, but ~30 % of events that the original Stage B included as bridge events fall *outside* every region's tighter radius in the variant. Trade-off: variant has more spatial purity, fewer events relocated. Not necessarily worse — depends on which question you're answering.

**5. Both Stage B catalogs flag ~97 % of relocated events as physical** (below seafloor depth). Jan 2019 v2 was only 92 % — the global-cluster solve pulled more events into shallow / above-seafloor positions. The Stage B partitioning is gentler on shallow events: they get located in the context of their immediate neighbours instead of being pulled toward the global swarm centroid. The 3 % residual "above seafloor" events in Stage B are the depth-sanity post-filter outliers.

### Cross-catalog comparison — known bug

I attempted to compare same physical events across catalogs (Stage A vs Stage B, Stage B vs Stage B variant, GrowClust vs Stage B) by joining on hypoDD's `id` column. **Those comparisons are currently invalid.** Root cause: `scripts/22_pyocto_to_hypodd_input.py` assigns `hypodd_id = row_index + 1` *after* `reset_index` on whatever pyocto subset it processes. So `id=1` in the full-year run is a different physical event than `id=1` in any `sub_X` run. The Stage B merge step (`scripts/26_merge_stage_b.py`) uses `id` to dedupe bridge events between sub-regions — meaning the "5,064 unique events" claim is likely an *overcount* of unique events, because the same pyocto event appearing in two sub-regions gets two different hypoDD ids and is not detected as a duplicate.

**Fix for next session:** change line 58 of `scripts/22_pyocto_to_hypodd_input.py` from
```python
ev["hypodd_id"] = (ev.index + 1).astype(int)
```
to
```python
ev["hypodd_id"] = (ev["event_idx"] + 1).astype(int)
```

then re-run scripts 22, 23, 24 for all Stage B sub-regions and re-merge. Final event counts will drop (real duplicates will collapse) but the catalog is then correctly indexed and cross-catalog comparison becomes meaningful.

**For now we can only claim within-catalog stats are valid.** The depth distributions, RMS residuals, and swarm geometry are correct; the event-count totals are an upper bound.

## Regional events / coverage gap

Question: did the events farther from the OBS network disappear from the relocated catalogs? Quantitative check:

| Source | Total events | At >60 km from any station |
|---|---|---|
| pyocto (input) | 42,040 | ~3,460 (≈8 %) |
| GrowClust year (relocated) | 11,033 | 194 |
| hypoDD Stage B | 5,064 | 0 |
| hypoDD Stage B-jan2019 | 3,579 | 0 |
| hypoDD Stage A | 676 | 0 |
| hypoDD Jan 2019 v2 | 5,610 | 0 |

**Every hypoDD catalog drops all 3,460 regional events.** GrowClust keeps ~6 % of them. Two causes:

1. **DD linking is intrinsically pair-coupled.** Regional events arrive at only a handful of stations and rarely satisfy MINLNK/MINOBS. They're dropped at the `ph2dt` stage and never enter the LSQR.
2. **Stage B sub-region radii (max 3 km from a backbone centroid) don't reach regional event locations.** The Stage A backbone is entirely inside the OBS array; K-means partitioning around it can't include far events.

This is *expected* DD behaviour, not a bug. The events still exist in `catalogs/pyocto_events_picker_only.csv` at their absolute-method (pyocto) positions. Maximum event-station distance in the pyocto catalog is 127 km -- everything is local-to-regional, no teleseismic.

### How to relocate them

Double-difference methods can't refine regional events; they need absolute-location methods that solve each event independently from its own picks:

1. **NLLoc (NonLinLoc)** -- probabilistic oct-tree, per-event posterior PDF. Already on the project roadmap (see `next_steps_after_full_year.md`). Best fit for the regional 8 % of the catalog.
2. **HYPOELLIPSE / HYPOINVERSE** -- classic deterministic absolute locators.
3. **Trust pyocto** -- pyocto's octree association is itself an absolute method, so its positions for regional events are already reasonable. No-effort fallback.

Recommendation: for the final catalog, tag each event with a `location_method` column (`pyocto`, `hypodd`, `growclust`) and use pyocto positions for events that no DD method relocated. Optionally run NLLoc as a one-pass cross-check for everything; this also calibrates absolute-frame drift in the DD result.

## Critical assessment of the swarm lineations

User observed that within the Jan 2019 hypoDD animation (keep_fresh = 3 days, so each pulse is visible only briefly), the swarm appears to trace **NW-SE lineations early in the swarm**, then **rotate to SW-NE** later. Asked whether this is geologically real or an inversion artifact. Senior-seismologist take:

Likely a mix:

- **NW–SE early phase: plausibly real.** Aligns with the regional Bransfield back-arc rift trend. Dike propagation in a magmatic intrusion typically follows pre-existing structural weaknesses, and NW–SE is the rift axis here.
- **SW–NE later phase: suspicious.** The OBS array is oriented roughly NW–SE, so hypoDD's resolution kernel is *systematically worse* perpendicular to the array. Marginal-quality events with fewer stations get smeared along the low-resolution direction, which is SW–NE. A 90° rotation in days is unusual for a real geological structure; it's exactly the signature of a station-coverage / pick-quality change.

Diagnostics needed to settle it (not done yet):

1. Does the lineation appear in **all three** Stage B/B-jan2019/C runs at the same times? If only one, partition artifact.
2. Does the line appear in both **map view AND depth cross-section**? A dike is planar; events on it align in 3D. A 2D-only line is a relocation kernel artifact.
3. **Migration velocity check**: plot event distance from cluster centroid vs origin time. Real dike propagation gives a linear front at ~0.1-10 m/s; random scatter is artifact.
4. **OBS station coverage timeline**: did stations drop in/out between the NW–SE and SW–NE phases? Changes in available stations change the resolution kernel.
5. Compare against GrowClust year-long catalog (different DD algorithm). Persistent across both → more credible.

The `strict_A/` filtered plots are the right diagnostic — if the SW–NE line disappears under the strict filter but the NW–SE line survives, that's strong evidence the second phase was marginally-coupled events all along.

## Senior-seismologist critique of the whole pipeline

For the record, on the question "which Stage B variant is most reliable":

- All three give the **same first-order answer**: swarm centroid (−62.42°, −58.38°), depth ~2-3 km below seafloor, footprint 5×5 km, peak Feb 2019. That conclusion is robust across configurations and against the GrowClust year-long catalog.
- **Stage B-jan2019** is the most reliable choice for the Orca swarm specifically (densest seed, tightest partition, most bridge events, smaller per-region LSQR problems).
- Stage B has broader spatial coverage including non-swarm satellites.
- Stage C is the most data-driven (elbow-method N) but at the cost of two large LSQR sub-problems and fewer bridge events.

What's **NOT** defensible from the current pipeline:

- Individual event positions at sub-km precision (no bootstrap).
- Sub-cluster geometry within the swarm (region-local cid IDs).
- Migration velocities or rates (no migration-front diagnostic done).
- Absolute lat/lon better than ~1 km (no NLLoc absolute reference).

For a publishable result, the gaps are:

1. **NLLoc** absolute reference catalog with per-event PDF uncertainties (the missing absolute frame).
2. **GrowClust3D.jl** as an independent DD relocator on identical input.
3. **hypoDD with `nboot=100`** for formal uncertainties.
4. **Pick-uncertainty calibration** on a subset against manual picks.
5. **Comparison to ≥1 published Bransfield catalog**.
6. **Velocity-model sensitivity test** (perturb Vp by ±5 %, re-relocate).

Until those are done, treat the Stage B/C catalogs as internal-consistency checks, not as the final answer. The first-order conclusion is solid; the precision claims and any lineation interpretation are not.

## Key lessons & decisions

1. **Year-long monolithic hypoDD is intractable on serial hardware** when one cluster has ~20 k events and ~8 M dt observations. LSQR convergence is single-iteration-bound and each iteration takes >11 h. Stop trying that path.
2. **The two-stage (pruned backbone + dense sub-clusters) approach is what's recommended in the field.** Same scientific question, dramatically tractable: ~10 min total wall on this catalog.
3. **Always capture stderr in subprocess wrappers** and don't impose a Python-level timeout on long-running Fortran jobs unless you really want them killed.
4. **Subprocess timeouts kill the binary, not just disconnect.** If you need to bail out without losing work, kill the *parent* (python) and the child (hypoDD) reparents to init and keeps running.
5. **K-means on a backbone catalog gives a fine spatial partition**, but radius-capping is essential -- otherwise one outlier point gives a region 22 km wide that swallows the swarm and reproduces the monolithic problem.
6. **Bridge events** (overlap between sub-regions) preserve inter-region coupling and let the merge step resolve conflicts cleanly. Each event is kept once, from the sub-region for which it is most interior.
