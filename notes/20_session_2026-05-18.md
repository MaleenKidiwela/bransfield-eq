## Build, patch, and silent-failure debugging

### NLLoc compile
`Makefile.ORIG` in upstream NonLinLoc was missing the `include/` path. Fixed
by building from `www_beta/src/Makefile` instead. Modern gcc (`-fno-common`
by default) tripped on duplicate-symbol errors; worked around with
`CC="gcc -fcommon"`. NLLoc, Vel2Grid, Grid2Time installed to
`~/.local/bin/`. Vel2Grid3D failed to link (more multiple-def errors); we
don't need it since we're reusing grids.

### Sign-stripped grid headers (took an hour to find)

The shipped `nlloc/time/ORCA.P.BRA*.time.hdr` files declare grid origin
`29.8 20.0 0`. But the `.buf` data was unambiguously built for origin
**(-29.8, -20.0, 0)**:

- Argmin of `ORCA.P.BRA05.time.buf` lies at (ix=158, iy=115, iz=0)
- Under origin `+29.8, +20.0` that maps to physical position (61.4, 43.0)
  km — far outside the array, totally inconsistent with the .hdr line-2
  station entry `BRA05 1.7355 3.0241 0`
- Under origin `-29.8, -20.0` it maps to (1.8, 3.0) km — matches the
  station entry exactly

Wrote `scripts/27_patch_nlloc_hdr.py` to rewrite the first header line
with the correct negative origin. Idempotent.

### Rotation sign error (caught only after seeing the plot)

In the NLLoc control file (`scripts/29_write_nlloc_control.py`) I wrote
`TRANS SIMPLE -62.4413 -58.44 -36`. The Stingray `srGeometry.rotation`
field is `+36` (CW from east).

Verified by hand-projecting BRA05 (lat=-62.4102, lon=-58.4472) into
Stingray frame with both signs:
- `rot=+36`: (1.7349, 3.0188) km → matches the `.time.hdr` station entry
  (1.7355, 3.0241)
- `rot=-36`: (-2.3349, 2.5829) km → garbage

First NLLoc run produced relocations that the user immediately spotted as
"rotated, and shown above the seafloor." Both symptoms came from the
double-applied wrong rotation (effective +72° error) pushing many events
onto the grid boundary. Fix was a 1-char diff: `-36` → `36`.

### Datum bug + rebuild

User caught it: **srModel is parameterized in depth-below-seafloor**,
not depth-below-sea-level. Two consequences:

1. NLLoc's output `depth_km` is already depth-below-seafloor — the
   "above seafloor" interpretation of shallow events was a plotting
   misread, not a physics bug. (After the rotation fix, those events
   were real shallow seismicity right under the OBS.)
2. My ORCA_v2 outer-ring padding came from
   `configs/velocity_model.csv`, which is sea-level-referenced
   (contains a 1.3 km water layer at top). That was a coordinate
   mismatch — inner block = depth-below-seafloor, outer ring =
   depth-below-sea-level — making events outside the srModel envelope
   locate with a 1.3 km vertical bias.

**Fix**: strip the water layer from `velocity_model.csv` (depths ≥ 1.3 km,
shifted so 0 = top of rock = seafloor) before padding. One-line edit in
script 38, rebuild SLOW_LEN, rerun Grid2Time × 38, rerun NLLoc.

### Lessons (worth remembering)

- **`-36` vs `+36`**: Stingray rotation is +36 CW. NLLoc TRANS SIMPLE
  takes RotCW the same way. One wrong sign = locations look rotated
  by 72° and pile up at grid edges. Verified by hand-projecting BRA05
  with both signs against the grid's .time.hdr station entry.
- **Sign-stripped grid origins** in shipped .hdr files (positive when
  they should be negative). Caught only by argmin-projection of the
  .buf data — the .hdr text is unsigned junk if you don't check.
- **srModel z is depth-below-seafloor**, not depth-below-sea-level.
  The shallow Vp values (1.7-1.9 km/s) are sediment, not water. If
  z=0 had been sea level we'd see 1.45 km/s (water) at the top.
  The 1D padding model's water layer must be stripped to keep the
  two datums consistent.
- **Modern gcc + old NLLoc**: `-fcommon` is required for both compile
  AND link. Plain `Makefile.ORIG` ships with paths assuming headers
  in `src/`, but on this clone they're in `src/include/`; the
  `www_beta/src/Makefile` works.
- **NLLoc paths**: LOCFILES is treated as cwd-relative even with a
  leading `/`. Don't use absolute paths in control files.
- **NLLoc has no seafloor constraint**: water is just a slow layer in
  the eikonal. Events can locate above the seafloor numerically.
  After the datum fix this becomes "events pile up at depth=0
  (cost-min at top of search grid)" instead — same underlying issue,
  different boundary symptom.
